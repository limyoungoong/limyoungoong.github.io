## ⚠️ Bootstrap (read this FIRST on a fresh machine)

This project's editorial direction, positioning, user profile, and writing-collaboration rules **do not live in this repo**. They live in an external memory folder synced via Dropbox, so the author's personal-branding strategy is never committed to a public GitHub repo.

**Before doing any content work** (writing posts, editing About, adjusting tone, suggesting layouts), verify the memory is connected:

```bash
ls -la ~/.claude/projects/*/memory/MEMORY.md 2>/dev/null
```

If `MEMORY.md` is missing or the `memory` path is a broken symlink, **stop and ask the user**:

> "이 프로젝트는 Claude 메모리를 Dropbox로 동기화합니다. 이 PC에서 Dropbox 루트 경로를 알려주세요 (예: `/mnt/d/Dropbox`, `~/Dropbox`, `D:\Dropbox`). 안에 `00. ★obsidian_note/90_System/Llm/limyoungoong.github.io/` 폴더가 있어야 합니다."

Then create the symlink (adjust `SLUG` for the current OS — it's the folder name under `~/.claude/projects/` matching this repo's absolute path with `/` replaced by `-`):

```bash
# WSL / Linux / macOS
DROPBOX="<user-provided path>"
SLUG="$(pwd | sed 's|/|-|g')"   # or: ls ~/.claude/projects/ to find it
rm -rf ~/.claude/projects/$SLUG/memory
ln -s "$DROPBOX/00. ★obsidian_note/90_System/Llm/limyoungoong.github.io" \
      ~/.claude/projects/$SLUG/memory
```

```powershell
# Windows (admin PowerShell, native Claude Code)
$dropbox = "<user-provided path>"   # e.g. "D:\Dropbox"
$slug = "<find-under-%USERPROFILE%\.claude\projects\>"
Remove-Item -Recurse -Force "$env:USERPROFILE\.claude\projects\$slug\memory"
New-Item -ItemType Junction `
  -Path "$env:USERPROFILE\.claude\projects\$slug\memory" `
  -Target "$dropbox\00. ★obsidian_note\90_System\Llm\limyoungoong.github.io"
```

Once linked, read `MEMORY.md` and the files it points to. **Do not write content** for this site without that context loaded — the editorial posture (practitioner field record, NOT junior portfolio) is non-obvious and easily violated.

The detailed setup guide (including conflict notes) lives at `$DROPBOX/00. ★obsidian_note/90_System/Llm/limyoungoong.github.io/SETUP.md`.

## What this repo is

Personal blog at `https://limyoungoong.github.io` built from the [Chirpy Starter](https://github.com/cotes2020/chirpy-starter) (Jekyll + `jekyll-theme-chirpy` gem). Deployed automatically to GitHub Pages via `.github/workflows/pages-deploy.yml` on every push to `main`.

## Common commands

Setup once: `bundle install`

| Task | Command |
|---|---|
| Local dev server (live reload on `127.0.0.1:4000`) | `bash tools/run.sh` |
| Local production build preview | `bash tools/run.sh -p` |
| Production build + html-proofer (mirrors CI) | `bash tools/test.sh` |

`tools/test.sh` is the canonical pre-push check — it runs `JEKYLL_ENV=production bundle exec jekyll b` then `htmlproofer _site --disable-external`. The CI workflow runs the equivalent inline.

## Architecture notes that matter

**Most of the site code is not in this repo.** Layouts, includes, sass, and core JS live inside the `jekyll-theme-chirpy` gem (see `Gemfile`, pinned to `~> 7.5`). Only the minimum override surface lives here: `_config.yml`, `_plugins/`, `_tabs/`, `_data/`, `_posts/`, `assets/`, `index.html`. If you need to read or override theme behavior, inspect the installed gem (`bundle info jekyll-theme-chirpy`) — don't expect to find templates locally.

**`assets/lib` is a git submodule** pointing to `cotes2020/chirpy-static-assets`. The deploy workflow currently has `submodules: true` commented out in `.github/workflows/pages-deploy.yml`, so the submodule contents are NOT pulled at build time. If a change ever requires those self-hosted assets at build time, uncomment that line. The submodule frequently appears as `m` (modified) in `git status` due to nested-checkout state — usually harmless; don't commit it unless you intentionally bumped the pin.

**Custom Jekyll behavior** is limited to `_plugins/posts-lastmod-hook.rb`, which sets each post's `last_modified_at` from `git log` when the post has more than one commit. This means accurate "last modified" timestamps depend on real commit history — squashing or rewriting history will change displayed dates.

**Content model**:
- `_posts/` — blog posts. Filename format: `YYYY-MM-DD-title.md`. Permalink is `/posts/:title/` (configured in `_config.yml`). Currently empty.
- `_tabs/` — sidebar nav pages (about, archives, categories, tags). They're a Jekyll collection sorted by front-matter `order:`.
- `_data/contact.yml`, `_data/share.yml` — sidebar contact links and post share buttons.
- Categories/tags archive pages are generated by `jekyll-archives` (see config block at bottom of `_config.yml`).

**Excluded from build** (see `exclude:` in `_config.yml`): `tools/`, `README.md`, `LICENSE`, `*.gem`, `*.config.js`, `package*.json`. So putting scripts in `tools/` keeps them out of the published site.

## Repo privacy boundary

The repo is **public** (GitHub Free doesn't allow Pages from private repos). Branding strategy, positioning copy, draft material, and any sensitive editorial work are kept out via two mechanisms:
1. `.gitignore` blocks LLM tool files: `CLAUDE.local.md`, `AGENTS.md`, `GEMINI.md`, `.claude/`, `.cursor/`, `.cursorrules`, `.aider*`, `.windsurfrules`.
2. The Claude memory (where editorial intent lives) is symlinked to an external Dropbox folder — never in the repo at all.

`CLAUDE.md` (this file) is intentionally committed because it only contains repo orientation + bootstrap instructions, no strategy content.
